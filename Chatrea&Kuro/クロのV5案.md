## 1. discrete burst 問題 — 「パ・パ・パ・パ」が出ない理由

### 現状の音の構造 (クロの推定)

```
tick:    |        |        |        |
         ↓        ↓        ↓        ↓
startloop  startloop  startloop  startloop
         |────────|────────|────────|
         ←   loop 再生 (持続)   →
```

`groove~` は `sig~ 1` が常時入力されていて、`loop 1` が有効。
つまり一度 `startloop` が来ると、**次の startloop まで loop 範囲を鳴らし続けている**。

`startloop` は「頭出し」であって「発音開始」ではない。
だから入れ子に聞こえる。`パパパパ` の粒が独立していない。

### 欲しいもの

```
tick:    |          |          |          |
         ↓          ↓          ↓          ↓
       パ___       パ___       パ___       パ___
        ↑ 短い burst  ↑ 沈黙  ↑ 短い burst
```

**各 tick が独立した短い burst** になってほしい。
tick 間には沈黙 or 急速な減衰があるべき。

### 解決案 (3つ)

#### 案A: envelope で絞る (クロの第一推奨)

```
[r tick_v4]
     |
     ├→ [startloop] → groove~ (頭出し)
     |
     └→ [line~ 1 10, 0 <burst_ms>] ← burst 用 envelope
              ↓
            [*~]  ← groove~ 出力にかける
              ↓
            dac~
```

**仕組み**:
- tick が来るたびに envelope が `1 → 0` と短時間 (burst_ms) で減衰
- burst_ms が tick 間隔より短ければ、tick 間に沈黙が発生
- groove~ 自体は裏で回り続けるが、音として聴こえるのは burst の時間だけ

**長所**:
- `groove~` の構造に触らない
- burst_ms を可変にすれば 「短いプツン」 から 「滑らかな減衰」 まで連続的に調整できる
- click は envelope の傾きで制御できる (0ms だと click 大、数ms だと click 弱)

**短所**:
- line~ の発火タイミングと startloop が厳密に同期する必要がある (deferlow などで調整が必要かも)
- 未確認: `line~ 1 10, 0 <burst_ms>` の構文の正確さ

**clroの推奨デフォルト値**: tick 100ms に対して burst_ms = 30〜50ms くらい

---

#### 案B: groove~ を pulse 的に stop/startloop

```
[r tick_v4] → [startloop] → 鳴り始め
                |
                ↓
             [delay <burst_ms>] → [stop] → 停止
```

**仕組み**: tick ごとに startloop → 少し待って stop、を繰り返す。

**短所**:
- stop の瞬間に click が出やすい
- 毎回 stop → startloop のオーバーヘッドで挙動が不安定かも
- deferlow 問題

→ **採用しない方がよさそう。案A の方が素直。**

---

#### 案C: window_ms を tick 間隔より短く設定するだけ

これは一番シンプル。今の構造のまま、`window_ms = 50ms`, `tick = 100ms` みたいな設定にすると、loop を1周鳴り切ってから次の startloop が来る。

**長所**: 追加オブジェクトゼロ
**短所**:
- wav の短い場所しか鳴らせない
- 音楽的に範囲を広げたいときに制約になる
- envelope の形は groove~ の素の音量のままで、尖らせたり丸めたりできない

→ **案A のサブセット。案A を採用すれば不要。**

### クロの推奨

**案A (envelope で絞る)** を採用する。
これなら `groove~` の持続を維持したまま、音として聴こえる粒を独立させられる。
burst_ms を ユーザー可変にすれば、「パ・パ・パ・パ」 から 「滑らかな follow」 まで連続的に移行できる。
これは xiroqs としても強い表現軸になる気がする。

## 2. 進行基準 — tick 基準 → wav 長基準

### チャトレアの整理を受けて理解した点

クロは `step_ms` を「何ms ずつ進むか」として扱っていた。これは tick 基準の発想。

でも欲しいのは、**wav (or 選択範囲) 全体を何等分するか** という発想。

```
tick 基準 (現状):
  step_ms = 200ms (固定)
  → wav が 10s でも 1s でも、毎回 200ms ずつ進む
  → wav 長さによって「何回で 1周するか」が変わる

wav 長基準 (欲しい):
  step_ratio = 1/16 (固定)
  → wav が 10s なら 625ms ずつ
  → wav が 1s なら 62.5ms ずつ
  → どんな wav でも「16回で 1周」
```

### これがなぜ有機的か

wav の長さは素材固有の「時間の長さ」。
それを基準にするということは、**trigger の機械性を保ちつつ、進行の基準は素材に委ねる**ということ。

役割分担:

| 要素 | 担当 | 性格 |
|---|---|---|
| a_time | **いつ**鳴らすか (timing) | 機械的 |
| 発火する瞬間 | **どこを**鳴らすか (position) | 素材基準 |
| wav length | **どれだけ進むか**の基準 (span) | 素材基準 |

これで「trigger は機械 / source は有機」の対比が、位置制御のレベルでも成立する。

### 実装案

```
[r tick_v4]
     |
     ↓
   [t b b]
    右(先) 左(後)
    |       |
    ↓       └→ [startloop]
   [f current_start]
    |
    ↓
   [+]  ← 右inlet に dynamic step_ms
    |
    ...

[r length_base]  ← wav_length or selection_length
     |
     ↓
  [* 0.0625]     ← step_ratio (= 1/16)
     |
     ↓
  dynamic step_ms
     |
     ↓ (進行ロジック内の [+] 右inlet へ)
```

### length_base の取得

**範囲選択していないモード**: wav 全体長
**範囲選択しているモード**: latch_end - latch_start

wav 全体長の取得方法:
- `buffer~` の右outlet に `info` メッセージを送ると、buffer の情報が出る (未確認、要 Max docs 確認)
- もしくは `info~` オブジェクト (未確認)
- 最悪の場合、ユーザーが flonum で手入力するフォールバック

ここは **実機で Max のリファレンスを確認してほしい領域**。
クロの推定は 50/50 で、docs での確認が必要。

### step_ratio の設計

- 典型的な値: 1/16, 1/32, 1/8
- ユーザーが flonum or slider で変更可能
- これが xiroqs の「時間の粒度」を決めるパラメータになる

## 3. 2モード設計

### モード定義

| モード | 材料 | base_length | 進行範囲 |
|---|---|---|---|
| mode 1: 素の追従スタッター | wav 全体 | wav_length | 0 → wav_length |
| mode 2: Loop型追従スタッター | 選択範囲 | selection_length | latch_start → latch_end |

### 内部構造

2つのモードは、「**base_length をどこから取るか**」の違いだけで、同じロジックで動く。

```
mode flag (0 or 1)
     |
     ↓
  [gswitch2]  ← 2入力を切り替え
    入力0 ← [r wav_length]
    入力1 ← [expr $f1 - $f2]  (latch_end - latch_start)
     |
     ↓
  base_length
     |
     ↓
  [* step_ratio] → dynamic step_ms
```

### reset ボタンの役割 (latch の代替)

reset を押すと:

1. 現在のモードに応じて current_start を初期化
   - mode 1: current_start = 0
   - mode 2: current_start = latch_start
2. loop_min / loop_max を再計算して groove~ に送る
3. 再生中なら stop → startloop で作り直し

### モード切替の検出

**案1: 明示的なトグルで切替**
- UI に「mode 1 / mode 2」のラジオボタン or トグル
- 切替時に自動で reset

**案2: 範囲選択の有無で自動検出**
- `selstart < selend` なら mode 2、そうでなければ mode 1
- でもこれだと「たまたま範囲選択したまま mode 1 に戻りたい」ができない
- 現時点では **案1 の明示的なトグル** が制御しやすい気がする

### latch は UI から消える

チャトレアの整理通り、latch ボタンは消して、内部処理として吸収。
mode 2 に切り替えた瞬間に latch 相当の処理 (selstart/end を内部に保存) が自動実行される。

## 4. sel 0 検討

現状: `sel 0 2` (4拍に 2回 = 拍頭と裏)
提案: `sel 0` (4拍に 1回 = 拍頭のみ)

これは discrete burst 問題と連動する。
tick 間隔が空くほど burst が独立して聴こえる。`sel 0` の方が疎だから、入れ子感が軽減されるはず。

**クロの推奨**: まず `sel 0` から始めて、discrete burst が成立してから `sel 0 2` に戻して密度を上げる手順でよさそう。

## 5. 統合した v5 の構造スケッチ

### 概念図

```
a_time (機械) ──┐
                ├→ いつ発火するか
wav length (源泉) ──┐
                    ├→ step_ms を動的に計算
                    ├→ base_length (mode で切替)
                    |
[current_start 進行] ←─────┘
       ↓
groove~ に loop 範囲 送信
       ↓
startloop (tick から)
       ↓
[line~ envelope] (burst 絞り込み)
       ↓
*~ で音量制御
       ↓
ezdac~
```

### 実装図 (層構造)

```
① selector UI         v3 と同じ
② a_time grid         v3 と同じ / sel 0 に変更検討
③ mode switch         mode 1 / mode 2 のトグル + base_length 切替 (gswitch2)
④ progression         base_length * step_ratio → dynamic step_ms
                      → current_start 進行 → loop_min/max 送信
⑤ reset               mode 切替時 or 明示ボタンで状態初期化
⑥ groove~ source      v4 と同じ
⑦ burst envelope      line~ で tick ごとに envelope 発火 → *~ で絞る
⑧ output              dac~
```

## 7. クロから逆に確認したいこと

### A. envelope の burst_ms は動的にしたい?

burst_ms を tick 間隔に対して固定比率で自動計算する案もある。
例: `burst_ms = tick_interval * 0.3` など。
これだと密度を変えても burst の「相対的な短さ」が保たれる。

どう思いますか?

### B. base_length を「ユーザーが手動 override できる」ようにする?

wav 全体長 / 選択範囲長を基準にするのが基本だけど、
「自分で 2000ms を基準にしたい」みたいな場面もあるかも。
その場合、flonum override を用意する?
それとも源泉 (wav) 以外を基準にするのは xiroqs の美学から外れる?

### C. reset は自動 or 明示?

- 自動: mode 切替時に必ず発火
- 明示: ボタンで発火
- 両方: mode 切替時に自動 + 明示ボタンもあり

xiroqs の「主体はてるちゃん」方針からすると、**両方** が自然かも。

## 一文要約

v4 で見えた 「入れ子問題」 は groove~ の持続再生と startloop のギャップが原因で、これを line~ による burst envelope で切り分ける。
step_ms は tick 基準ではなく wav length 基準に変え、「trigger は機械 / 進行基準は素材」の対比を位置制御に持ち込む。
2モード設計は base_length の取得元切替で統一的に実装し、latch ボタンは reset に置き換える。
