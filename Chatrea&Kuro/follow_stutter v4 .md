# follow_stutter v4 相談メモ

## 現状の理解
現行の v3 は、

- `[metro 100]`
- `-> [a_time 15 4]`
- `-> [sel 0 2]`
- `-> [startloop]`

という構成で、`groove~` に対して**同じ選択範囲を再侵入**している。

つまり現状は、

- パパパパの trigger は機械的に成立している
- ただし、読む位置は固定
- 選択範囲の内部は前向きに読まれるが、wav 全体の中を進行しているわけではない

この理解でいる。

---

## 次にやりたいこと
次は、**パパパパの機械的 trigger はそのまま維持**したまま、

- trigger のたびに
- `loop start`
- `loop end`

を少しずつ前へ進めたい。

つまり、

**固定範囲の再侵入**
ではなく、

**移動範囲の再侵入**
にしたい。

---

## 欲しい挙動
理想の挙動はこう。

1. `a_time -> sel 0 2` で発火
2. 発火ごとに `current_start` を更新
3. `current_end = current_start + window_ms`
4. その updated な範囲を `groove~` に送る
5. その後に `startloop`

これにより、

- パパパパの区切り感は残す
- ただし毎回同じ場所ではなく
- wav 上を少しずつ前へ進みながら再侵入する

状態にしたい。

---

## 重要な前提
ここで主役なのは `waveform~` ではない。  
主役は **a_time による時間制度** である。

- source / 演奏 = 有機的
- trigger / 再侵入タイミング = 機械的

この対比は維持したい。

そのため、
**追従は常時連続でヌルヌル更新するのではなく、`sel 0 2` の節目でだけ更新**したい。

---

## やりたいことを Max 的に言い換えると
必要なのは、おそらく以下のような段。

- `current_start` を保持する仕組み
- 発火ごとに `current_start += step_ms`
- `current_end = current_start + window_ms`
- その値を `groove~` の loop start / end へ送る
- その後で `startloop`

---

## 相談したい点
以下の観点で、Max 実装として筋の良い最小構成を考えてほしい。

1. `a_time -> sel 0 2` を維持したまま、移動範囲再侵入へ拡張する方法
2. `current_start` / `step_ms` / `window_ms` の持ち方
3. `loop start/end` 更新と `startloop` の順序
4. 実在する Max オブジェクトだけで組むこと
5. 概念図と実装図を分けること

---

## 一文要約
**v3 は固定範囲のパパパパ再侵入。  
v4 では、a_time によるパパパパは維持したまま、読む範囲だけを少しずつ前へ進めたい。**
