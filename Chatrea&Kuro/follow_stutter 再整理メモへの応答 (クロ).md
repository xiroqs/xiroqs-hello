## 0. 最初に押さえる 1 点 — クロが出した仮説

3つの問いに答える前に、クロが考えた核の仮説を先に置きます。

**「固定位置の気持ち良さ」を保ったまま追従するには、追従の単位を「tick」ではなく「a_time の 1 周」にすればよい。**

つまり:

- tick ごとに進めると (v4 で試した構造)、固定位置が 1 tick しか続かず、retrigger の気持ち良さが壊れる
- a_time の 1 周ごとに進めると、固定位置で数回 retrigger した後に次の位置へ移る
- 「a_time の 1 周」は、a_time が時間制度の中核である以上、音楽的に意味のある単位

この仮説が成立するなら、「固定位置 × 追従」は 同一構造の 2 層で両立できる。

## 1. 概念図 — 3 層構造

```
┌──────────────────────────────────────────┐
│ 層1: 時間制度 (変わらない骨格)            │
│ metro + a_time                           │
│  → 分割された ms の格子                    │
└──────────────────────────────────────────┘
              │
              ├──────────────┬──────────────┐
              ↓              ↓              ↓
       ┌─────────────┐ ┌──────────────┐
       │ 層2:         │ │ 層3:          │
       │ 固定位置     │ │ 1周単位の     │
       │ retrigger    │ │ 進行           │
       │              │ │               │
       │ 毎 tick or   │ │ a_time 1周    │
       │ 拍頭で      │ │ ごとに        │
       │ startloop    │ │ current_start │
       │              │ │ += step_ms    │
       └─────────────┘ └──────────────┘
              ↓              ↓
         固定位置の       次の固定位置へ
         パパパパ         移動 (追従)
```

### 層の役割

| 層 | 役割 | 速度 |
|---|---|---|
| 層1 | 時間を等しい ms で分割する骨格 | 静的 (演奏中は不変) |
| 層2 | 各 tick で retrigger を送る | 速い (tick 単位) |
| 層3 | a_time 1周ごとに位置を進める | 遅い (1 周単位) |

### ここでの発見

**層2 と 層3 は速度が違う。**
層2 が速いからこそ固定位置の気持ち良さが成立し、
層3 が遅いからこそ進行がメロディに従う感覚が生まれる。

もし層2 と層3 の速度が同じだと (= tick ごとに進行)、
固定位置は存在せず、just a moving window になってしまう。
これが v4 で感じた違和感の正体だと思う。

### a_time の再定義

この 3 層構造における `a_time` は、
「機械的な格子」というだけでなく、**「1周で音楽的な 1 単位 (= 1小節相当) を切る」道具**でもある。
つまり a_time には 2 つの役割がある:

1. 各 tick の位置を出力する → 層2 の retrigger に使う
2. 1周の終わり (or 始まり) を検出する → 層3 の進行トリガに使う

この 2 役割は、同じ a_time からの出力を分岐するだけで両立できる。

## 2. 問いへの回答

### 問い1. fixed-position retrigger の核を保ったまま追従をどう足すか

→ **追従の発火を「a_time 1周ごと」に限定する**。

具体的には、a_time の出力を分岐して:

- `sel 0` 系 → startloop に送る (固定位置 retrigger)
- `sel 15` 系 → 進行 bang として送る (1周ごとに 1 回発火)

これだけで、固定位置 retrigger と追従進行が同じ格子の上に共存する。

### 問い2. 追従は tick ごとに進むだけでよいのか、音素材の流れを参照する必要があるのか

→ **音素材の流れを参照する必要はない。** a_time の機械的制度の上で進めばよい。

理由:

- 音素材を解析して追従すると、「trigger は機械 / 素材は有機」という xiroqs の対比が崩れる
- a_time という時間制度を中核に据えている以上、追従もその制度の上で動くのが筋
- 「メロディラインに沿う」という感覚は、素材解析から生まれるのではなく、
  **進行の単位が音楽的である** ことから生まれる

ただし、「音素材の流れに引きずられる」のではなく「音素材の上に新しい時間を被せる」という
チャトレアの整理からは、音楽的単位 (= a_time 1周) を持つだけで十分と考えた。

### 問い3. 最小構成の追従版はどういうものか

→ **v4 progression の構造から、以下 1 点だけ変える**:

| v4 | v5 |
|---|---|
| 進行は **毎 tick** で発火 | 進行は **a_time 1周ごと** に発火 |

他は v4 と同じ (`current_start` 保持、`+ step_ms`、`if` 折り返し、
`t f f f` で loop_min / loop_max 送信)。

変わるのは進行の発火契機だけ。
これが「最小構成」の意味。

## 3. 実装図 (actual patch blueprint)

### 構造全体

```
① selector UI (v3/v4 と同じ)
② latch (内部処理、UI ボタンなし)
③ a_time grid (v3/v4 と同じ)
④ retrigger 層 (層2 = 固定位置)
⑤ progression 層 (層3 = 追従進行)  ← ④と⑤を a_time から分岐
⑥ groove~ source
```

### ③ a_time grid の改造

v3/v4 では `a_time 15 4` → `sel 0 2` → `startloop` だった構造を、
出力を 2 方向に分岐する:

```
[toggle]
    ↓
[metro 100]
    ↓
[a_time 15 16]           ← 0 〜 15 の int を出力 (1 周 = 16 tick)
    ↓
[t i i]                  ← int を 2 outlet に分配
 左      右
 ↓       ↓
[sel 0]  [sel 15]
 ↓       ↓
 retrigger   progression
 (層2へ)     (層3へ)
```

**重要な設計判断**: a_time の引数は `15 4` ではなく `15 16` に変更する。
理由: 1周で 16 tick 進んでから戻る方が、「1小節 = 16 等分」の音楽的直感に近い。
% の値 (4) は拍位置を分けるためのものだったが、
固定位置 retrigger に `sel 0` しか使わないなら、% 16 で全 tick を通す方がシンプル。

ただしこれは **要確認点**。
てるちゃんの耳で「`a_time 15 4` の方が気持ち良い」という感触があれば、そちらを優先する。

**代替案**: `a_time 15 4 → sel 0` にしておいて、1周検出は a_time の別方法で作る。
例えば counter を並走させるなど。ただし複雑になるので、まずは `a_time 15 16` 案から試したい。

### ④ retrigger 層 (層2)

```
[sel 0] (a_time 分岐 左)
    ↓
[startloop]
    ↓
[send startloop_v5]
```

### ⑤ progression 層 (層3)

v4 の進行ロジックをほぼそのまま再利用。
入力 bang を `receive tick` から `receive progression_bang` に変えるだけ。

```
[sel 15] (a_time 分岐 右)
    ↓
[send progression_bang]


[r progression_bang]
    ↓
[t b b]
 右(先) 左(後)
 |       └→ (層2 と独立なので startloop は送らない)
 ↓
[f current_start]
    ↓
[+ 0.]  ← 右inlet: flonum step_ms
    ↓
[if $f1 > $f2 then $f3 else $f1]
  $f2 ← r latch_end_v5
  $f3 ← r latch_start_v5
    ↓
[t f f f]
 右(先)  中   左(後)
 |       |    └→ [+ 0.] ← flonum window_ms
 |       |          ↓
 |       |       [send loop_max]
 |       └→ [send loop_min]
 └→ [f current_start] 右inlet (保存)
```

### v4 との差分サマリー

| 項目 | v4 | v5 |
|---|---|---|
| a_time の引数 | `15 4` | `15 16` (要確認) |
| sel の使い方 | `sel 0 2 → startloop` | `[t i i] → sel 0 + sel 15` |
| 進行の発火契機 | 毎 tick | a_time 1周ごと (sel 15) |
| 進行と retrigger の同期 | t b b で順序保証 | **独立** (層2 と層3 は別経路) |
| startloop | 進行後に送信 (t b b 左出力) | 層2 から毎 tick 送信 |

**重要な変更**: v4 では「進行 → startloop」が t b b で順序保証されていた。
v5 では進行と retrigger が別経路になるため、順序保証は不要になる。
代わりに、進行のタイミングが a_time 1周ごとに限定されるので、
「ちょうど 1周の終わりで位置が更新され、次の tick の retrigger で新しい位置が鳴る」
という時系列が自然に成立する。

## 4. 気になる点 / 未確認事項

### A. a_time 15 16 の挙動

- `15 16` で counter 0 15 → % 16 が本当に 0〜15 の全値を出力するか
- もし 0〜15 を全部通すなら OK だが、実機で確認したい

### B. sel 15 で 1周検出ができるか

- `[sel 15]` に int 15 が来たら bang を出す、標準動作のはず
- もし問題があれば `[sel 0]` で「0 の瞬間 = 前の周が終わった直後」を検出する代替もある

### C. 進行と retrigger の時系列

- sel 15 (progression) で loop_min/max が更新される
- 次の tick は sel 0 (retrigger) で startloop が発火
- この 2 つの間 (metro の 1 tick 分 = 100ms) で、
  groove~ の loop 範囲更新が間に合うか
- Max の message 処理は通常 sample-level より速いので、問題なさそうだが要確認

### D. 「メロディラインに沿う」の確認

- 層3 が 1周ごとに step_ms 進むだけで、本当にメロディラインに沿って聞こえるか
- これは実機で聴いて判断する領域
- step_ms と window_ms の比率、metro の ms と a_time の分割数の組み合わせで
  音楽的感触がかなり変わるはず

### E. ms 基準の維持

- window_ms, step_ms はすべて flonum で直接操作
- wav 長は一切参照しない (前のメモで下げたまま)
- この方針はチャトレアの整理通り

## 5. クロから確認したいこと

### Q1. a_time の引数について

`15 4` と `15 16` のどちらがよいか、てるちゃんの耳の直感はどう?

`15 4` は「4拍」の感覚が強く、`15 16` は「16分音符の連続」の感覚。
固定位置 retrigger + 追従を両立するなら `15 16` の方がシンプルだが、
`15 4` でも `[t i i]` の右側を別の周期で分ければ可能。

### Q2. progression の発火頻度

a_time 1周ごと (= 16 tick に 1 回) で、実際に「メロディラインに沿う」感覚が出るか。
もし「もっと速く進みたい」「もっと遅く進みたい」という感触があれば、
`sel 15` を `sel 7 15` にして 1周に 2回発火させる、などの調整が可能。

### Q3. window_ms と step_ms の関係

- window_ms > step_ms → loop 範囲が重なる (オーバーラップ)
- window_ms < step_ms → loop 範囲に隙間ができる (スキップ)
- window_ms = step_ms → 隣接

クロの予想では、window_ms > step_ms の「少し重なる」状態が
メロディラインに沿う感覚を作りやすい気がする。

## 6. 一文要約

固定位置 retrigger は層2 として維持し、追従進行は層3 として a_time 1 周ごとに発火する
独立経路で実装する。
a_time の出力を `[t i i] → sel 0 / sel 15` で分岐するだけで、同じ時間制度の上に
固定位置と追従の 2 層が共存できる。
v4 progression の構造はほぼ再利用し、変えるのは「発火契機を毎 tick から 1 周ごとに」
という 1 点のみ。
