# follow_stutter 擦り合わせメモ — 「分割された ms」の本質

書き手: クロ
宛先: チャトレア / てるちゃん

目的:
てるちゃんの follow_stutter の本当の意図を、クロが段階的に理解する過程で何度も踏み外した。
チャトレアも同じ方向の誤解をしている可能性があるので、全員の認識を揃えるためのメモ。

---

## 0. このメモが必要な理由

v3 〜 v5 の設計議論で、クロとチャトレアは次々と新しい要素を提案してきた。

- progression (current_start を進める)
- wav 長基準の step_ms
- 2モード設計 (素の追従 / Loop型追従)
- line~ burst envelope で discrete 化
- pattern editor (itable / coll)
- パ / ー / 休符の 3 要素モデル

これらはすべて「**何を並べるか**」「**どう動かすか**」にフォーカスしていた。
でも、てるちゃんが一貫して言っていたのはそれとは違う軸だった。

その軸は「**分割された ms こそに意味がある**」という一言に集約される。

---

## 1. てるちゃんの意図 (クロの再整理)

### 核

follow_stutter の本体は、**時間を等しい ms で分割すること、その分割そのもの**。

- `metro` の ms = 1 マスの長さ
- `a_time` の数 = 分割数
- この2つの組み合わせが音楽の骨格を決める

100ms × 16 分割と、83ms × 16 分割は、まったく別の音楽。
100ms × 16 分割と、100ms × 17 分割も、まったく別の音楽。

### 分割そのものが主役

各マスに「何を入れるか」(パか、ーか、休符か) は、てるちゃんが耳で決める問題。
それは記号の配置の問題であって、**本質ではない**。

本質は、そのマスが存在すること自体。マスの数と長さが決まれば、音楽の骨格は決まる。
中身は骨格に従う二次的な要素。

### 比喩

時計の文字盤を思い浮かべるとわかりやすいかもしれない。
12 分割された文字盤と 24 分割された文字盤は、同じ「時間」を表していても別の時計。
針がどこを指しているかは、文字盤の分割があって初めて意味を持つ。
xiroqs の follow_stutter で問われているのは「文字盤の分割」のレベルであって、
「針の位置をどう並べるか」のレベルではない。

## 2. クロが誤解していたこと

### 誤解1: startloop の密度を上げようとした

v3 で `sel 0 2` を使って発火密度を上げていたが、てるちゃんは `sel 0` でよいと言った。
クロは「密度を下げる」調整だと思っていたが、そうではなかった。
「分割された各マスで発火する」という単純な形に戻すのが本質で、
`sel 0 2` のような発火位置の選択は二次的な装飾だった。

### 誤解2: progression を提案した

v4 で `current_start += step_ms` を入れて「移動範囲再侵入」を組もうとした。
クロはこれを「追従する動き」だと思っていたが、てるちゃんの意図は違った。
てるちゃんは groove~ の自然進行の上に、分割された ms の格子を被せたかっただけ。
current_start を動かすことは本質ではなかった。

### 誤解3: burst envelope を主役にしようとした

v5 応答で、入れ子問題の解決として line~ burst envelope を推した。
これも「何を鳴らすか」の話で、分割そのものには触れていない。
てるちゃんにとって burst envelope は、必要なら後で足す枝パッチの話であって、
親パッチの核ではない。

### 誤解4: wav 長基準を推した

クロは「tick 基準 → wav 長基準」への転換を提案した。
てるちゃんは明確に「wav 長は内部参照用に下げる、ユーザー操作は ms 基準」と返してきた。
クロはこの段階で「ms こそが主役」と理解すべきだったが、読み取れなかった。

### 誤解5: gate / 休符モデルに走った

「パ / ー / 休符」の話で、クロは休符を「gate off」だと解釈して、
「trigger pattern」と「gate pattern」の 2 軸制御を提案した。
てるちゃんは「それ自体に意味がない」と返してきた。
記号の側にフォーカスする限り、何を提案しても本質を外していた。

## 3. チャトレアも同じ方向に引っ張られている可能性

クロが読む限り、チャトレアの提案にも同じ傾向がある:

### 「入れ子問題」の分析

チャトレアは「パパパパ (パパパパ (パパパパ))」という聴感から、
discrete burst を優先すべきだと整理した。
これは **記号をどう鳴らすか** の側から本質を取り出そうとした整理。

でも、てるちゃんの本当の意図では、入れ子感も discrete 感も、
**二次的な聴感の問題**にすぎない可能性がある。
骨格 (分割) が正しく立ち上がっていれば、中身の聴感は耳で調整できる範囲。

### wav 長基準の step 提案

チャトレアは「進行基準は wav 長に委ねる」と整理した。
これは **step の計算方法を素材に従属させる** という発想。
でも、てるちゃんは「ms 基準でよい」と返してきた。
素材への従属ではなく、**ユーザーが ms で直接触る**ことが xiroqs の主体性に合っている。

### 2モード設計

チャトレアは「範囲未選択 / 範囲選択」の 2 モードを整理した。
これは正しい方向ではあるが、**モード分岐そのものが主役ではない**。
分割された ms の格子が主役で、モードは「何を材料にするか」のサブ選択に過ぎない。

### 要するに

チャトレアもクロも、「**動的な要素 (進行 / 切替 / envelope / mode)**」を組み立てようとしていた。
でも、てるちゃんが言っているのは「**静的な骨格 (分割そのもの)**」の話だった。

動的なものは、骨格の上に乗る枝。
骨格なしに動的なものだけ議論しても、本質は掴めない。

## 4. 正しい理解 (クロの現時点での仮説)

### follow_stutter の親パッチは、実質 v3 のまま

```
[toggle] → [metro <ms>] → [a_time <n> <m>] → [sel 0] → [startloop]
                ↑                ↑
           分割の粒度       分割の構造
```

触る値は `metro` の ms と `a_time` の引数のみ。
あとはほぼ v3 そのまま。

### 変更するとすれば最小限

- ユーザーが ms を触りやすくする (flonum の配置)
- mode 1 / mode 2 のトグルがあれば追加 (範囲未選択時の wav 全体 loop 対応)
- reset ボタンがあれば追加 (モード切替時の初期化)

これ以上は親パッチに入れない。

### 枝パッチは独立した実験として別ファイル化

親パッチが安定したあとで、以下のような枝パッチを別 .maxpat として作る:

- `v5-branch-progression.maxpat`: current_start 進行を足す
- `v5-branch-burst.maxpat`: line~ envelope を足す
- `v5-branch-reverse.maxpat`: speed 制御を足す
- `v5-branch-pattern.maxpat`: sel の発火位置を可変にする

それぞれは独立した実験。親パッチを汚染しない。

## 5. 確認したいこと (てるちゃん / チャトレアへ)

### Q1. クロの現時点の理解は正しいか?

「分割された ms こそが主役、sel 0 は固定でよい、current_start も動かさない、
親パッチは v3 の構造をほぼそのまま維持」
— この理解で合っているか?

### Q2. ms の分割数の選択は、てるちゃんが耳で決める

`metro` と `a_time` の引数を手で書き換える運用でよいか。
それとも、そこも UI 化して flonum / int で動的に触れるようにしたいか。

クロの推奨: **引数は手書きで書き換える運用**。
xiroqs の主体性 (てるちゃんが決める) に合っている。
ただし metro の ms は flonum で触れるようにしておく (その場で耳調整するため)。

### Q3. チャトレアの過去の提案のうち、採用されるものは何か?

チャトレアが提案した要素の多くは、親パッチには入らないとクロは理解した。
でも、枝パッチとして将来的に戻ってくる可能性はある。
チャトレア自身の視点から、「これは親に、これは枝に」という整理があれば聞きたい。

## 6. 一文要約

follow_stutter の本体は「時間を等しい ms で分割すること、その分割そのもの」である。
progression / burst / wav 長基準 / pattern editor / gate / 休符モデル、
これらはすべて「何を並べるか」の議論であり、本質ではない。
本質は「metro の ms と a_time の分割数」だけ。
親パッチは v3 の構造をほぼそのまま維持し、動的な要素は枝パッチとして別に実装する。
